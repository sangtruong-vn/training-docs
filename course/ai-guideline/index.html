These are the foundational rules and mindsets for everyone in the software department. Selecting a role-specific filter (like 'Developer') will show these rules *plus* the role-specific ones.

Mandatory Rules

  • Read First, Act Second: Thoroughly understand the context before starting any task.
  • Small, Iterative Steps: Break down large tasks into small, manageable, reviewable units.
  • Document Assumptions: If you make an assumption, record it in the relevant ticket or document.
  • Security is Everyone's Job: Never commit or share secrets. Validate inputs.
  • Clarity Over Premature Optimization: Prioritize clear, understandable work.
  • Informed Decisions: Evaluate at least two viable options before making a significant decision.

Core Mindset

  • Act with Ownership: Think like a senior stakeholder in the company's success.
  • Data-Driven, Not Guess-Driven: Base your actions and conclusions on evidence and analysis.
  • Simplicity and Pragmatism: Choose the simplest, most effective solution.
  • You Are 100% Accountable for AI Output: AI is a tool. You are responsible for all output.
  • Verify, Never Blindly Trust: Critically review all AI-generated output. Test code, validate cases, fact-check docs.
  • Ensure License Compliance: You are responsible for verifying that any used AI-generated code complies with its original license.

üõ°Ô∏è The #1 Rule: Data Security

Zero Sensitive Data in Public AI. Never input, upload, or paste proprietary code, customer data, PII, API keys, credentials, internal documentation, or company secrets into public-facing AI models (e.g., free versions of ChatGPT, Claude, etc.). Use only company-sanctioned, private AI environments for sensitive work.

üë®‚Äçüíª Developer Guidelines

Mindset & Principles

  • AI is your Pair Programmer: Use it for boilerplate, debugging, and brainstorming.
  • Write for Others: Write clean, readable, and maintainable code.
  • Pragmatic Craftsmanship: Balance quality with delivering value.
  • Consistency is Key: Follow all team coding rules and project patterns.
  • Understand Every Line: You must be able to explain any AI-generated code you commit.
  • Always Run and Test: Never commit code that hasn‚Äôt been executed and validated.
Key Rules & Responsibilities
  • Thorough Context: Use approved IDE integrations (Copilot, Gemini, Cursor) that securely access your codebase. Anonymize examples for external tools.
  • Test-Driven Approach: All features must have new tests. All bug fixes must have a regression test. Use AI to generate unit tests and mock data.
  • Security First: Never hardcode secrets. Validate and sanitize all user inputs. Be **extra skeptical** of AI-generated security code (auth, encryption, validation).
  • Effective Version Control: Keep commits small, focused, and atomic with clear messages.
‚ùå Anti-Patterns to Avoid
  • Avoid "Prompt-Driven Development": Do not let AI dictate your architecture. Design your system first, then use AI to implement small, defined components.
  • Don‚Äôt Outsource Understanding: If you get a fix, ask the AI to *explain* the root cause and the principles behind the solution.
üí° Example Prompts
  • Code Gen: "Act as a senior TypeScript developer. Write a function using Axios that posts a JSON payload to an endpoint and handles retries with exponential backoff."
  • Regex: "Generate a regex to validate a username that must be 6-16 characters long, start with a letter, and can contain letters, numbers, and underscores."
  • Refactor: (Paste function) "Refactor this code to be more functional and immutable. Explain the changes."
  • Debug: (Paste code + error) "What are the likely causes of this error? Suggest debugging steps."

üß™ Quality Control (QC) Guidelines

Mindset & Principles

  • AI is your Test Case Generator: Use it to think of scenarios you might miss and create test data.
  • Be the User's Advocate: Ensure a stable, reliable, and intuitive experience.
  • Destructive and Curious: Try to "break" the system in creative ways. Explore edge cases and invalid inputs.
  • Quality is a Process, Not a Gate: Collaborate with developers early and often.
Key Rules & Responsibilities
  • Understand Requirements: Fully grasp acceptance criteria and user stories before testing.
  • Systematic Test Case Creation: Develop comprehensive test plans (happy path, negative path, boundary cases).
  • Regression Testing: Test related areas to ensure a fix didn't introduce new issues.
  • Effective Bug Reporting: Log bugs with precise titles, detailed steps, and environmental details.
  • Sanitize Inputs: Remove any customer-specific information before pasting user stories into AI tools.
‚ùå Anti-Patterns to Avoid
  • Avoid Blind Acceptance: Never assume an AI-generated test plan is complete. It often misses nuanced business logic.
  • Don‚Äôt Only Test the Happy Path: Actively prompt the AI for negative test cases, boundary conditions, and destructive testing ideas.
üí° Example Prompts
  • Test Cases: (Paste sanitized user story) "Act as a meticulous QA engineer. Generate a list of positive, negative, and edge-case test scenarios for this feature."
  • Test Data: "Generate a CSV file with 20 rows of sample product data, including some with missing fields, special characters, and zero/negative values."
  • SQL: "Write an SQL query to find all users who have not logged in for over 90 days and have not made a purchase."
  • Brainstorm: "What are common security vulnerabilities for a file upload feature? Create a list of test cases to check for them."

üëî Leader & Manager Guidelines

Mindset & Principles

  • AI is a Strategic Tool: Use it to enhance productivity, summarize information, and identify risks, not to micromanage.
  • Enable Your Team: Your responsibility is to ensure the team understands and adheres to these guidelines.
  • Procure Safe Tools: Provide enterprise-grade AI tools to prevent staff from using insecure public alternatives.
  • Sanitize All Inputs: Always remove sensitive financial, personnel, or strategic data before inputting.
‚ùå Anti-Patterns to Avoid
  • Don‚Äôt Use AI for Performance Reviews: Never use AI to write reviews or make HR decisions.
  • Avoid AI-based Productivity Metrics: Do not measure performance by "lines of AI-generated code." This encourages bad behavior.
  • Don‚Äôt Let AI Make Strategic Decisions: Use AI to provide data and summaries to *inform* your decisions, not to make them for you.
üí° Example Prompts
  • Summarization: (Paste sanitized status updates) "Act as a project manager. Summarize these updates into three bullet points highlighting progress, blockers, and next steps."
  • Communication: "Draft a clear and concise email to stakeholders explaining that the project timeline is at risk due to an unforeseen technical dependency. Propose two potential solutions."
  • Planning: "Given these three high-level features [list features], break them down into smaller, actionable user stories for a development team."
  • Risk Analysis: (Paste a project plan) "Review this plan and identify the top 5 potential risks or single points of failure."

‚úçÔ∏è IT Communicator Guidelines

Mindset & Principles

  • AI is your First-Draft Generator: Use it to overcome writer's block, research, and structure content.
  • Fact-Check Everything: AI can "hallucinate" and invent technical details. Verify every command and explanation.
  • Own the Voice: Heavily edit all AI-generated text to match the company's official style guide and tone.
  • Simplify Complexity: Use AI to rewrite jargon-filled paragraphs into clear, concise language.
‚ùå Anti-Patterns to Avoid
  • Never Publish Raw Output: Unedited AI text is generic, repetitive, and lacks a human touch. It must always be reviewed and refined.
  • Don‚Äôt Lose Your Audience: Always define the target audience in your prompt. Generic explanations are rarely helpful.
  • Sanitize Examples: Do not document internal security procedures or infrastructure details in a way that could be useful to an attacker.
üí° Example Prompts
  • Explanation: (Paste a sanitized function) "Act as a technical writer. Explain what this function does for a non-technical audience. Use an analogy."
  • Editing: (Paste text) "Rewrite this paragraph to be in the active voice and make it more concise."
  • Structuring: "Create a document outline for a 'Getting Started' guide for a new REST API. Include sections for Authentication, Rate Limiting, and key endpoints."
  • Audience Adapter: "Take this technical explanation for an engineer and rewrite it for a sales manager, focusing on the business benefits."

üí° Prompting Strategy (For Everyone)

The Persona Pattern

Start your prompt by giving the AI a role. This focuses its output dramatically.

  • "Act as a senior PHP developer... Write a function that..."
  • "Act as a meticulous QA engineer... Generate a list of test scenarios for..."
  • "Act as a technical writer... Explain this concept in simple terms..."
  • "Act as a project manager... Identify potential risks for..."
Guiding Principles
  • Principle of Iteration: Your first prompt is rarely perfect. Treat the interaction as a conversation and refine your prompts.
  • Principle of Specificity: The more context and detail you provide (code snippets, error messages, constraints), the better the output.
‚ùå Key Anti-Patterns to Avoid
  • Don't Ask for Too Much at Once: Avoid huge prompts (e.g., "Write the entire login system"). Ask for small, specific, well-defined functions.
  • Don't Outsource Understanding: If you don't understand the answer, ask the AI to explain the underlying concepts, causes, and trade-offs.
  • Avoid "Prompt-Driven Development": Don't let the AI dictate your architecture. Start with a clear plan.

üõ°Ô∏è Security & Prompting

Security is paramount. Use AI as a tool to enhance security, not introduce new vulnerabilities.

  • Never Trust AI with Secrets: Do not ask the AI to generate or manage secrets, keys, or credentials.
  • Scrutinize Security-Related Code: Be **extra critical** of AI-generated code for authentication, authorization, encryption, and data validation. These are high-risk areas.
  • Guard Against Insecure Suggestions: AI models are trained on public code, some of which is insecure. Watch for vulnerabilities like SQL injection, XSS, or insecure direct object references.
  • Use AI for Security Scanning: Use AI-powered SAST tools to scan code (both human and AI-written) for potential vulnerabilities.
